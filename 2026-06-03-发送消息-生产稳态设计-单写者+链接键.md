# 发送消息 · 生产稳态设计（单一写者 + 确定性链接键 + recents 防抖）

> 状态：**定稿设计（待评审，未实施）** ｜ 日期：2026-06-03
> 取代/精化前稿 `2026-06-03-A方案-发送即本地落权威行-设计稿.md`：在确认
> **D1=「发送方会收到自己这条 SESSION_SUMMARY_UPSERT」** 后，**不再走双写 + 本地合成排序键**，
> 改走**单一写者 + 链接键**——更稳、无排序分歧、blast 更小。
> 关联分支：仍建议在 `fix/subscribe-resync-decouple-p2p4` 合入后实施。

---

## 1. 设计取向（为什么最稳）

| 维度   | 选择                                                                                                   | 收益                                                      |
| ------ | ------------------------------------------------------------------------------------------------------ | --------------------------------------------------------- |
| 写者   | **单一写者**：权威消息行只由 push applier 写                                                           | 无双写竞态、解耦、守住阶段2                               |
| 排序   | **服务端 `sort_key` 全程权威**，不引本地排序键                                                         | **无本地≠服务端分歧**；与前端「单调插入＋回服务端序」一致 |
| 配对   | **链接键 `clientMsgId↔localMessageId`**，保证权威出站行恒带 `request_message_id`                       | 确定性配对，删文本/附件启发式，无双行/挪位                |
| 列表   | recents **本地即时写预览（"你好"/"[图片]"）+ 置顶信号**（仿草稿，**不抬版本键**）；push 版本门权威对齐 | 预览/位置都瞬时丝滑、不踩 LWW 坑、push 仍可纠正           |
| 持久化 | in-flight 出站**不本地落行**（等 push 落库），崩溃窗口靠 reconcile 兜底                                | 换取最小复杂度与零分歧；详见 §6 取舍                      |

**核心不变量**：**权威出站行恒带 `request_message_id = clientMsgId`**——由「push 回填」与「send 兜底 update」双向保证（§4），与业务后台是否回带该字段无关。

---

## 2. 解耦分层（谁拥有什么）

```
UI 层      乐观气泡（瞬时, key=clientMsgId）—— 只管即时反馈，可丢可重建
链接层     clientMsgId ↔ localMessageId —— send resp 学到，后端 OutboundLink 持有（内存有界）
真相层     权威消息行（PK=localMessageId, ORDER BY 服务端 sort_key）—— 唯一写者 = push applier
列表层     recents：本地置顶信号(瞬时, 仅进 ORDER BY) + 权威 summary(push, 版本门 LWW)
```

各层只通过**键**耦合，不共享写路径——这是「解耦」的落点。

## 4. 关键不变量如何被双向保证

权威出站行的 `request_message_id` 在两种到达顺序下都被填上（`request_message_id` 在
`upsert_messages` 的 `ON CONFLICT` 冻结，故只需保证**首次落行**或**事后专项 update**其一命中）：

- **resp 先于 push（常态）**：send 已 `OutboundLink.put(L→C)` → push applier 落首行时回填。✅
- **push 先于 resp（极端）**：push 落首行 `reqId=''` → send 的**专项 update**（`WHERE reqId=''`）补上 + 发通知 → 前端重读配对。✅
- **业务后台原生回带 reqId**：上述全成冗余安全网，更稳。✅

## 5. 边角鲁棒性

| 场景                                | 行为                                                                                      |
| ----------------------------------- | ----------------------------------------------------------------------------------------- |
| push 乱序（PENDING/CONFIRMED 倒序） | 同 L 幂等 UPSERT，`send_status` 非倒退合并，终态收敛                                      |
| push 早于 resp                      | §4 专项 update 兜底；最坏短暂双行后由 reqId 收敛                                          |
| reqId 缺失                          | 链接键回填保证不变量                                                                      |
| 发送失败                            | 不写行（单写者）；气泡 failed；重发复用 C 幂等；recents 信号可选清                        |
| 离线                                | E①（已落）：横幅 + 发送置灰；不产生悬挂请求                                               |
| 崩溃/重启（push 未回）              | in-flight 出站未本地落行 → 丢失；下次打开会话 reconcile 从服务端补回（§6 取舍）           |
| 清空重拉                            | 删本地行 → refetch 用服务端 sort_key 重灌 → 与服务端完全一致                              |
| 本地预览 vs 服务端 summary 文案     | 文本一般逐字一致(无闪)；后端若对 summary 特殊格式化 → ~400ms 时一次极小文案对齐(通常无感) |

## 5b. 排序与抖动（关键·红线）

前端已落地**单调插入**（`chatStore.replaceAuthoritative`, 第十二轮，code 已核实 chatStore.ts:278-302）：
已显示气泡**不按服务端序重排**、新消息进底部、**收敛的乐观气泡锚定在原位**；冷开/重开退化为纯服务端序。
**本设计依赖且绝不回退这套机制**——一旦改回「每次按服务端序重排」抖动立刻回来。

| 排序点                 | 行为                                                  | 抖动                                              |
| ---------------------- | ----------------------------------------------------- | ------------------------------------------------- |
| 发出 → 乐观气泡入底    | 单调插入到底部                                        | 无                                                |
| push 权威版回来        | 按 `request_message_id` **确定性配对 → 锚定原位替换** | 无（确定配对让锚定可靠，根治旧「图片挪一下」）    |
| 发送时左栏             | `local_last_sent_at_ms` 置顶，`MAX(...)` 不回落       | 无（消除旧「草稿掉落 下→上」抖动）                |
| push summary 回来      | 位置已在顶、文案相同 → 无可见变化                     | 无                                                |
| 并发入站               | 单调插入入底，已显示不重排                            | 无（本地序可能≠服务端 = **静态**差异，重开纠正）  |
| push 早于 resp（边角） | 一瞬无键可对 → 短暂双行 → reqId/serverId 收敛         | 极短 transient（边角；后端原生回带 reqId 则为 0） |

**结论：不会有严重抖动。** 本设计**移除**当前两处真实抖动（图片挪一下、草稿掉落），只余一个边角 transient。
残留的「本地序 sub-second ≠ 服务端」是**静态**差异（无动画/闪烁，重开纠正），不是抖动。

## 6. 取舍：in-flight 出站不本地持久化

- **代价**：发送成功但 push 回来前崩溃/退出 → 该消息不在本地缓存（重开 reconcile 补回，窗口小、可恢复）。
- **为何接受**：换来「单写者 + 零排序分歧 + 最小 blast」。若「绝不丢」成为硬需求，再加 §7 的可选持久化（退化为双写，需接受 §例子2 的本地序或给 `upsert_messages` 加 sort_key 覆盖）。**默认不做。**

## 7. 改动清单（都小、分层）

**后端**

1. `OutboundLink`：新增内存有界 `local_message_id → request_message_id` 映射（仅撑 send→push 窗口，无需持久化）。
2. `MessageSync.send_message`：`hub.send_message` 成功后 →
   `OutboundLink.put` + 专项 backfill update（+ 条件发 ConversationMessages 通知）
   - recents **本地写预览(summary/type/direction) + 置顶信号 `local_last_sent_at_ms`**（**不抬 `last_message_sort_key_ms`**）+ 发 RecentSessions 通知。
3. `message_event.rs` decode/apply：`requestMessageId` 空时用 `OutboundLink.get(L)` 回填。
4. recents 迁移：加客户端列 `local_last_sent_at_ms`（或复用 `opened_at_ms` 语义），并入 `list_top` 的 `ORDER BY MAX(...)`；远端 UPSERT 永不触碰（仿 `local_draft_at_ms`）。

**前端** 5. `chatStore.replaceAuthoritative`：以 `request_message_id == clientMsgId` 为**主**确定性配对；
保留 serverId 兜底；**分步移除** `authTextMatchesOptimistic`/`authAttachmentMatchesOptimistic`/`matchedEcho` 启发式（每步留回归测试）。

> 不动 push 回灌主链、不动水位门、不引本地排序键、不双写权威行。

## 8. 丝滑时间线（用户所见，无抖动）

```
t0        点发送        → 气泡(sending)立刻在底部；会话立刻置顶 + 左栏预览立刻"你好"  (全本地)
t0+ε      resp 回        → 气泡 "已发"
t0+~400ms push 回        → 气泡原位换权威版(无双行/无挪位)；左栏 summary 权威确认(同文案,无可见变化)
                          排序/预览全程不回退/颠倒/闪烁
```

## 10. 一页结论

- 最稳来自**少做**：单写者不变、排序全程服务端、只加一个**确定性链接键**和一个**本地置顶信号**。
- 丝滑来自**三段无抖**：乐观气泡即时 → 已发 → 权威原位替换；左栏发送即置顶、预览版本门补全。
- 解耦来自**分层只靠键**：UI/链接/真相/列表四层互不共享写路径。
- 与前端既有「单调插入＋回服务端序」(第十二轮) 同向，落地阻力小。
