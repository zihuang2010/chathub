## 1. 背景与现状链路

当前「发送一条出站消息」是**写读解耦**架构（阶段2 决策）：

```
前端 handleSend → 乐观气泡(sending) → sendMessage()
  → Tauri send_message (lib.rs:659)
    → MessageSync.send_message (message_sync.rs:404)
        a. ensure_window         // API 调用前先建保守空窗
        b. hub.send_message(...) // gRPC → relay forward → 业务后台
        c. 直接返回 resp，【不本地写消息行】
  → 前端 markSent(clientMsgId, resp.localMessageId)  // 气泡转 sent + 钉 serverId

业务后台异步把这条消息经 subscribe 流推回:
  SEND_PENDING_CREATED → SEND_CONFIRMED (均 MESSAGE_UPSERT, 同 localMessageId, sendStatus 递进)
    → apply_push_batch (message_event.rs:169): 热会话门控 → decode_message_row → upsert_messages
      → extend_window_newest → 发 ConversationMessages ChangeNotice
        → 前端重读本地缓存 → chatStore.replaceAuthoritative 收敛乐观气泡
```

**核心约束（阶段2）**：消息行的**唯一写者**是 push applier（`MESSAGE_UPSERT`），幂等靠主键
`local_message_id`。`send` 不写行、不 bump 水位、不发 ChangeNotice。

### 关键代码事实（已核实）

1. `SendMessageResp`（`hub.rs:871`）只有三字段：`local_message_id` / `send_status` /
   `message_time`（`"yyyy-MM-dd HH:mm:ss"` 服务端本地时区）。**没有 `sortKey`、没有完整 message 快照。**
2. `decode_message_row`（`message_event.rs:77`）强制要求 `message.sortKey` 非空，否则返回 `None`
   走兜底——**`sortKey` 是服务端权威字段，落行的硬前提**。
3. `MessagesStore.upsert_messages`（`messages.rs:76`）的 `ON CONFLICT(local_message_id) DO UPDATE SET`
   **只更新** `message_direction / content_text / send_status(不倒退) / attachments_json /
gmt_modified_time / revoked(黏) / fail_reason / updated_at_ms`；
   **不更新** `sort_key / message_time_ms / message_type / wecom_account_id / request_message_id`。
   → **先落的行其 `sort_key` 是终态，后到的 push 无法纠正排序。**（A 的头号风险）
4. `send_status` 合并是非倒退的（`0/1→任意`、`2→3/4`、`4→3`、`3 终态`）。
   → send 先写 `send_status=1`（已发送），push `SEND_CONFIRMED` 的 `3` 能正常升级。✅
5. 已存在 `upsert_message_and_bump_window`（`messages.rs:146`）：**单事务**「UPSERT 行 + 推进/新建水位窗」，
   注释明确为消除「行已落库但水位未 bump → 并发 reconcile 把刚发的行删掉」的危险中间态。
   **全仓无生产调用方（仅自身单测在用）= 为本方案备好的休眠原子原语。** ✅

## 2. 要解决的两个问题

### 问题 1：乐观↔权威收敛竞态（一整类前端 bug）

出站消息有两条命——①乐观气泡（前端即时）②权威行（push 回灌后重读）——靠
`chatStore.replaceAuthoritative` 三层配对收敛：
`requestMessageId==clientMsgId` → 附件 `objectName` → 纯文本内容。
启发式之所以存在，是因为权威重读（读本地缓存，极快）可能抢在 `markSent` 钉 serverId 之前落地，
且 push 回显的 `requestMessageId` **有时为空**（业务后台 `SEND_PENDING_CREATED` 不一定回带）。
历史上的「瞬时双行 / 图+文挪位」都源于此。

### 问题 2：持久化缺口

`send` 不写本地行 → 出站消息在 push 回灌落库前**只活在内存乐观 store**。push 落地前重启 App →
这条消息不在本地缓存（要等下次打开会话 reconcile / 历史拉取找回），存在一个丢失窗口。

---

## 3. 目标 / 非目标

**目标**

- G1：`send` 成功后，出站消息**立即落本地权威行**（不等 push 绕回），补上持久化缺口。
- G2：让权威出站行**确定性携带 `request_message_id = clientMsgId`**，从而前端可**纯靠 id 配对**，
  逐步**删除文本/附件启发式匹配**，根治问题 1 的 bug 类。
- G3：保持 push 回灌幂等收敛不变（同 `local_message_id` 升级 `send_status`、补 `attachments` 等）。

**非目标**

- 不改 push 回灌链路本身（subscribe/resync/水位门）——那是另一条在飞分支。
- 不追求删除乐观气泡（in-flight 期间仍需即时气泡；见 §7.1）。
- 不改业务后台契约为**前置条件**（若能改是加分项，见 §6 D1）。

## 4. 核心设计

在 `MessageSync.send_message`（`message_sync.rs:404`）的 `hub.send_message` 返回成功后，**新增一步**：
用 `resp` + 入参构造一条 `MessageRow`，调用**休眠原语** `upsert_message_and_bump_window`
原子落行 + bump 水位，再发一条 `ConversationMessages` ChangeNotice。

```
send_message:
  a. ensure_window                       // 不变
  b. resp = hub.send_message(...)        // 不变
  c. 【新增】row = build_outbound_row(resp, 入参, client_msg_id)
     store.upsert_message_and_bump_window(row, external_user_id, freshness_ms)  // 原子
  d. 【新增】change_notice_tx.send(ConversationMessages, {employee_id, conversation_id})
  e. 返回 resp                            // 不变
```

`build_outbound_row` 字段来源：
| 字段 | 来源 |
|---|---|
| `local_message_id` | `resp.local_message_id` |
| `request_message_id` | **`client_msg_id`（=前端 clientMsgId，确定性！）** |
| `conversation_id` / `wecom_account_id` / `employee_id` | send 入参 |
| `message_type` | send 入参 |
| `content_text` | send 入参 |
| `attachments_json` | 由入参 `file_path(objectName)/file_name/file_size/duration` 拼出**降级附件**（无 `mediaId`/`transferStatus`，push 后升级） |
| `send_status` | `resp.send_status`（通常 1=已发送） |
| `message_time_ms` | `parse_server_time_to_ms(resp.message_time)` |
| `sort_key` | **见 §5（linchpin）** |
| `message_direction` | 2（出站，本地方向） |
| `revoked` / `fail_reason` / `gmt_modified_time` | 缺省 |

**幂等收敛**：之后 push 的 `SEND_PENDING_CREATED/SEND_CONFIRMED`（同 `local_message_id`）经
`upsert_messages` 升级 `send_status`、补 `content`/`attachments`（含真实 `mediaId`/转存态）。✅

## 5. Linchpin：`sort_key` 从哪来（三选项）

`resp` 没有 `sortKey`，但落行必须有，且 `upsert_messages` **不会用 push 的 sortKey 纠正先落行**（§1.3）。
故 `sort_key` 的取法是本方案最关键的决策：

### 选项 A2（推荐做 MVP）：合成 sort_key，接受其为终态

合成 `sort_key = "<message_time_ms>-<local_message_id>"`，对齐服务端格式（`split_sort_key_ms` 取前导 ms）。

- 出站消息**即当前最新**，合成键前导 ms = 发送时刻 → 排在最末（最新）= 正确位置。
- push 到达时 `upsert_messages` 跳过 `sort_key` → 合成键保留。对「刚发的最新消息」几乎必然正确。
- **风险**：若服务端真实 sortKey 会把它排到别处（出站最新场景极罕见），则永久轻微错序。
- **成本**：纯本地，**零后端依赖**，最低风险。

### 选项 A3（加固，可选后续）：让 push 纠正 sort_key

在 `upsert_messages` 的 `DO UPDATE SET` 增加 `sort_key = excluded.sort_key`（及 `message_time_ms`），
使 push 的权威 sortKey 覆盖合成键。

- **收益**：A2 的错序被 push 自愈。
- **风险**：改动的是**全仓最关键的合并 SQL**，blast radius 大（reconcile / 翻页 / 所有 upsert 路径都过它）；
  且需确认 push sortKey 恒优于本地（理应如此）。建议**仅在 A2 观察到错序后**再上，且单独评审。

### 选项 A1（理想，需跨团队）：扩展 send 响应契约

业务后台 send 响应体加 `sortKey`（理想再加完整 message 快照）→ send 直接落**完全权威**行。

- **收益**：无合成、无降级附件、无 §5 风险。
- **成本**：跨团队契约变更（D1），不作为前置条件。`SendMessageResp` 加字段 + `decode` 对齐即可。

> **推荐路线**：A2 落地 MVP；A1 若业务后台能配合则最优；A3 作为 A2 的兜底加固，独立评审。

## 7. 影响面

### 7.1 前端乐观气泡：保留，但配对简化

- in-flight 期间（send 网络往返）仍需乐观气泡，**不删**。
- 但权威出站行现在**确定性带 `requestMessageId = clientMsgId`** → `authRequestIdMatchesOptimistic`
  永远命中。

### 7.2 可删除的启发式（G2 的兑现，依赖 D3）

`chatStore.ts` 中 `authTextMatchesOptimistic` / `authAttachmentMatchesOptimistic` /
`matchedEcho` 那段「权威回显抢在 markSent 前落地」的双行竞态兜底——在 A 落地后，
出站权威行恒有 `requestMessageId`，理论上可**逐步移除**（需保留充分回归测试再删，分步进行）。

### 7.3 与 C（整会话重读）耦合

A 让 `send` 也发 `ConversationMessages` ChangeNotice → 每次发送多一次整窗重读。
当前重读零网络 + `replaceAuthoritative` 已有内容等价短路，影响有限；但若叠加 C 一起评估更稳。

### 7.4 阶段2「单一写者」回退

A **明确回退**「消息行唯一写者 = push applier」为**双写者（send + push）**，靠 `local_message_id`
主键幂等 + `send_status` 非倒退合并收敛。这是有意识的架构取舍，需在代码注释/本稿留痕，避免后人误判。

## 9. 分阶段实施（A2 路线）

1. **后端落行（最小闭环）**

## 10. 验证方案

**后端（Rust 单测 / e2e）**

- `send_message` 后 `hub_conversation_messages` 立即有该 `local_message_id` 行，`request_message_id == clientMsgId`，`send_status==1`。
- 模拟 push `SEND_CONFIRMED` 同 id → `send_status` 升到 3、`attachments_json` 升级为带 mediaId/转存态；`sort_key` 行为符合所选 A2/A3。
- 原子性：并发 reconcile 不删除刚 `upsert_message_and_bump_window` 的新行（复用该原语既有 `*_atomic` 测试范式）。
- 失败路径（D4）：本地 upsert 失败仅告警、不影响返回 resp。

**前端（vitest）**

- 发送后经 ChangeNotice 重读即得权威出站行，乐观气泡按 `requestMessageId` 确定性替换、无双行。
- （阶段2 后）移除启发式不回归「图+文挪位 / 瞬时双行」既有用例仍绿。

## 11. 一页结论

- A 的真正价值：**补持久化** + **把权威出站行的 `requestMessageId` 收归我方掌控**，从而消灭一整类
  乐观↔权威收敛 bug；而非「降低 sending 时延」（happy-path 气泡早在 forward 返回即转 sent）。
- 技术上**可行且有现成原子原语**（`upsert_message_and_bump_window` 休眠待用）。
- 最大风险是 **`sort_key` 来源**（`upsert_messages` 不纠正先落行）→ MVP 走 A2 合成键，A3/A1 兜底/升级。
- **回退阶段2 单一写者**，需有意识接受。
- **实施时机**：等 `subscribe-resync-decouple` 分支落地后再动，避免热点文件冲突。
