# 发送接口与 notify 乱序客户端处理说明

## 背景

客户端调用发送消息接口后，服务端会同步返回发送结果；同时，服务端还会通过工具网关 notify-push 下发消息、会话摘要等异步通知。

由于 notify 是异步链路，客户端可能先收到接口响应，后收到 `SEND_PENDING_CREATED`、`LAST_SEND_STATUS_CHANGED`、`SEND_CONFIRMED` 等通知。客户端需要按字段合并状态，不能简单按到达顺序覆盖本地数据。

## 客户端处理规则

### 1. 点击发送后先建本地临时消息

客户端生成 `requestMessageId`，并立即在本地插入一条临时消息：

| 本地字段                               | 建议值                            |
| -------------------------------------- | --------------------------------- |
| `clientTempId`                         | 等于 `requestMessageId`           |
| `requestMessageId`                     | 本次请求 ID                       |
| `wecomAccountId`                       | 当前会话企微账号                  |
| `externalUserId`                       | 当前会话外部联系人                |
| `messageType`、`contentText`、附件字段 | 本次请求内容                      |
| `sendStatus`                           | 本地待发送态，可按服务端 `1` 处理 |
| `localMessageId`                       | 空                                |
| `sortKey`                              | 本地临时排序值                    |

### 2. 发送接口返回后绑定服务端消息 ID

用客户端保存的 `requestMessageId` 找到本地临时消息，合并接口响应：

| 响应字段         | 合并方式                           |
| ---------------- | ---------------------------------- |
| `localMessageId` | 写入本地消息，作为后续主匹配字段   |
| `sendStatus`     | 更新发送状态，但要遵守“不倒退”规则 |
| `messageTime`    | 更新消息时间                       |

接口响应没有 `sortKey`、`platformMessageId`、`platformSeq`，这些字段等待 notify 或历史接口补齐。

### 3. 收到 MESSAGE_UPSERT 后合并消息

消息匹配优先级：

1. `localMessageId`
2. `requestMessageId`
3. `platformMessageId`
4. `wecomAccountId + externalUserId` 仅作为弱兜底，不建议常规依赖

同一条消息的稳定字段可补齐：

| 字段                | 合并说明                                      |
| ------------------- | --------------------------------------------- |
| `platformMessageId` | 本地为空时补齐；发送成功通知中应有            |
| `platformSeq`       | 本地为空时补齐；发送成功通知中应有            |
| `sortKey`           | 用于消息排序，notify 到达后替换本地临时排序值 |
| `messageTime`       | 用服务端时间修正本地时间                      |
| `attachments`       | 按附件快照刷新附件展示和转存状态              |
| `failReason`        | 发送失败时展示失败原因                        |

### 4. sendStatus 合并不能倒退

聚合消息发送状态当前为：

| 值  | 含义     |
| --- | -------- |
| `1` | 待发送   |
| `2` | 发送中   |
| `3` | 发送成功 |
| `4` | 发送失败 |

客户端合并规则：

| 当前状态 | 收到状态      | 处理                               |
| -------- | ------------- | ---------------------------------- |
| `1`      | `2`、`3`、`4` | 接受                               |
| `2`      | `3`、`4`      | 接受                               |
| `3`      | `1`、`2`      | 忽略状态倒退                       |
| `3`      | `4`           | 原则上忽略；可触发历史消息刷新校验 |
| `4`      | `3`           | 接受成功，说明服务端后续确认成功   |
| `4`      | `1`、`2`      | 忽略状态倒退                       |

客户端判断发送状态时应以 `sendStatus` 为准，不要只依赖 `eventReason`。`eventReason` 用于说明触发原因和调试日志。

### 5. SESSION_SUMMARY_UPSERT 合并摘要

摘要匹配优先级：

1. `conversationId`
2. `wecomAccountId + externalUserId`

摘要更新规则：

| 判断                                          | 处理                                                                          |
| --------------------------------------------- | ----------------------------------------------------------------------------- |
| `incoming.lastSortKey > current.lastSortKey`  | 接受 `lastMessageSummary`、`lastSendStatus`、`lastMessageTime` 等最后消息字段 |
| `incoming.lastSortKey == current.lastSortKey` | 合并 `lastSendStatus`，遵守发送状态不倒退规则                                 |
| `incoming.lastSortKey < current.lastSortKey`  | 不覆盖最后消息展示；可合并联系人资料类字段                                    |
| `UNREAD_CHANGED`、`MARK_READ`                 | `unreadCount` 以服务端值为准                                                  |

### 6. notifySeq 用于通知去重和断点

客户端按 `employeeId + clientId` 维度保存最后已应用的 `notifySeq`：

| 判断                                    | 处理                                       |
| --------------------------------------- | ------------------------------------------ |
| `notifySeq <= lastAppliedNotifySeq`     | 重复通知，忽略                             |
| `notifySeq == lastAppliedNotifySeq + 1` | 正常应用                                   |
| `notifySeq > lastAppliedNotifySeq + 1`  | 中间缺通知，先拉补偿或历史消息，再继续应用 |

同一个 batch 内按 `events` 顺序应用。应用成功后再 ACK。

## 当前结论

现有代码字段可以支撑客户端处理发送接口响应与 notify 乱序问题。客户端实现时重点遵守：

1. 用 `requestMessageId` 关联接口响应前的本地临时消息。
2. 用 `localMessageId` 作为 notify 到达后的主匹配字段。
3. 用 `notifySeq` 做通知去重、顺序和断点判断。
4. 用 `sendStatus` 做发送状态判断，并禁止状态倒退。
5. 用 `sortKey`、`lastSortKey` 做消息和摘要的新旧判断。
6. 接口响应没有的平台字段，等待 notify 或历史接口补齐。
