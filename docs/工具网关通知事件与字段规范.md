# 工具网关通知事件与字段规范

本文档是企微聚合聊天“业务后台 -> 聚合工具网关 relay -> 客户端”的实时通知协议权威规范。客户接待列表同步策略见同目录 [客户接待列表轻量同步方案.md](客户接待列表轻量同步方案.md)，最近接待摘要维护机制见 [system-design.md](system-design.md) 的“最近接待会话摘要维护”章节。标准消息、摘要和 outbox 的本地事务一致性以 [消息摘要Outbox原子一致性机制.md](消息摘要Outbox原子一致性机制.md) 为准。

通知协议保持平台无关：业务后台只能推标准业务事件和平台中性字段。payload 不得出现 Easy `cmd/msgType/msgData/guid/syncMsg` 等专属结构；确需表达平台来源时，仅使用 `platformCode/platformAccountId/platformMessageId/platformSeq` 等平台适配层字段。

通知 payload 必须保持平台中性，只承载标准业务事件和业务快照。Easy `cmd/msgType/msgData/guid/syncMsg/msgUniqueIdentifier/seq` 等专属字段不得透传到工具网关或客户端；当前默认 `EASY` 平台的差异由业务后台平台适配层转换为本文的 `eventType/eventReason/message/sessionSummary/friend/accountBinding/accountStatus/forceClose` 字段。

消息关键词过滤规则见 [消息关键词过滤规则.md](消息关键词过滤规则.md)。客户新消息命中过滤关键词时，业务后台不生成该新消息对应的通知 batch，不占用 `notifySeq`，不向 relay 推送 `MESSAGE_UPSERT` 或由该消息导致的 `SESSION_SUMMARY_UPSERT`；标准消息仍入库，客户端主动查历史时仍可看到。

## 1. 协议边界与角色职责

### 1.1 三方职责

| 角色                           | 职责                                                                                                          | 不负责                                       |
| ------------------------------ | ------------------------------------------------------------------------------------------------------------- | -------------------------------------------- |
| `wechat-business-app` 业务后台 | 生成通知 batch、分配 `notifySeq`、持久化 outbox 逻辑盘、重投未确认 batch、支持工具网关按 `notifySeq` 补偿拉取 | 不维护客户端长连接，不直接推客户端           |
| 聚合工具网关 relay             | 接收 batch、幂等入库、fan-out、记录 ACK、断线重放、本地 relay event log 缺失时主动从业务后台 outbox 拉取      | 不生成或重排 `notifySeq`，不改写业务事件语义 |
| 聚合客户端                     | 维护一个员工级长连接、处理 batch、ACK、按事件局部更新 UI、异常兜底刷新                                        | 不作为权威数据源，不自行推算未读数或消息顺序 |

### 1.2 连接与路由模型

本期采用：

```text
employeeId
  -> clientSessionId / deviceId
    -> connectionId
      -> subscribedWecomAccountIds[]
```

- 一个员工客户端会话维护一个 HTTP/2 gRPC 长连接、一个订阅。
- 客户端收到员工级通知后，按 `wecomAccountId/conversationId` 自行 fan-out 到对应企微账号页签、接待列表项或会话窗口。
- `wecomAccountId` 只是企微账号订阅和业务定位维度，不能作为唯一连接维度或续点游标维度。
- 工具网关不得信任客户端声明的 `employeeId/wecomAccountId`；连接身份必须来自 `verifyToken` 返回值。

工具网关可以维护企微账号辅助订阅：

```text
wecomAccountId -> employeeId / clientSessionId / connectionId
```

辅助订阅用于收到某个企微账号消息时快速找到在线员工连接，但它不能替代 `employeeId` 主索引。真正是否允许推送，仍以业务后台返回的员工和账号绑定关系、连接上下文中的 `employeeId` 为准。

只按 `wecomAccountId` 路由会有以下问题：

- 员工管理多个企微账号时，客户端登录态是员工级，通知却被拆成账号级，容易漏掉跨账号的客户接待列表刷新。
- 企微账号禁用、换绑或转移后，旧员工连接可能继续收到该账号通知。
- `markRead`、发送结果、摘要刷新、通知 ACK 缺口都是员工客户端视图，不适合只挂在某个 `wecomAccountId` 上。
- 新好友、账号配置变更、强制下线等事件可能没有稳定的单个会话消息，不宜只用企微账号作为唯一路由键。
- 同一个 `wecomAccountId` 在绑定变更瞬间可能对应新旧两个员工视图，只按账号路由无法表达权限切换边界。

## 2. notifySeq 生成规则与 scope

`notifySeq` 由业务后台生成，聚合工具网关 relay 不重新编号。

推荐 scope：

```text
clientId + employeeId
```

本期固定：

```text
clientId = rh_wxchat
```

因此实现时必须把 `clientId` 纳入唯一键和幂等键。由于本期只有 `rh_wxchat` 一个客户端，业务含义上表现为每个员工一条递增通知流，但落库和 relay 判断仍使用 `clientId + employeeId`。

生成规则：

```text
notifySeq = nextSeq(clientId, employeeId)
```

`nextSeq(clientId, employeeId)` 本期使用 MySQL 序列表 + 行级锁实现，序列表为 `b_wecom_chat_notify_seq`。不要通过 `select max(notify_seq) + 1` 从 outbox 反推下一个序号：并发下容易撞号，表变大后查询成本也会不可控。

序列表逻辑字段：

| 字段               | 说明                                     |
| ------------------ | ---------------------------------------- |
| `client_id`        | 客户端标识，本期固定 `rh_wxchat`         |
| `employee_id`      | 目标员工 ID                              |
| `owner_shard_no`   | 员工分片号，默认 `employeeId % 64`       |
| `current_seq`      | 当前已分配并成功落 outbox 的最大通知序号 |
| `last_batch_id`    | 最近一次生成的 batch 幂等 ID             |
| `last_notify_time` | 最近一次生成通知时间                     |

唯一键：

```text
uk_client_employee(client_id, employee_id, is_deleted)
```

分配算法：

```text
begin transaction

ensureSeqRow(clientId, employeeId):
  insert ignore into b_wecom_chat_notify_seq(
    client_id, employee_id, owner_shard_no, current_seq
  ) values (
    clientId, employeeId, employeeId % 64, 0
  )

seqRow = select current_seq
         from b_wecom_chat_notify_seq
         where client_id = clientId
           and employee_id = employeeId
           and is_deleted = 0
         for update

notifySeq = seqRow.current_seq + 1
batchId = clientId + ":" + employeeId + ":" + notifySeq

insert into b_wecom_chat_notify_outbox_{ownerShardNo}(
  client_id, employee_id, owner_shard_no,
  notify_seq, batch_id, protocol_version, source_app,
  trace_id, event_count, event_types, payload_json,
  send_status, retry_count, retention_expire_time
) values (
  clientId, employeeId, employeeId % 64,
  notifySeq, batchId, "1.0", "wechat-business-app",
  traceId, eventCount, eventTypes, payloadJson,
  0, 0, now + retentionDays
)

update b_wecom_chat_notify_seq
set current_seq = notifySeq,
    last_batch_id = batchId,
    last_notify_time = now
where client_id = clientId
  and employee_id = employeeId
  and is_deleted = 0

commit
```

事务边界：

完整事务边界、幂等、失败处理和测试门禁见 [消息摘要Outbox原子一致性机制.md](消息摘要Outbox原子一致性机制.md)。本节只保留 `notifySeq` 分配的通知协议侧约束。

- `select ... for update` 只锁定同一个 `clientId + employeeId` 的序号行，不同员工通知流可并行生成。
- `current_seq` 更新和 outbox 插入必须在同一个本地事务内提交；事务回滚时序号不前进，未落库的 `notifySeq` 不对外可见。
- 每个会产生客户端可见通知的 outbox 必须写入稳定 `business_event_key`；重复回调、重复补偿和修复任务应先按 `uk_business_event` 复用已有 outbox，不得重新生成 `notifySeq`。
- 事务提交后才允许调用工具网关 `notify/push`。若提交成功但进程在推送前宕机，后台重投任务按 `send_status=0` 扫描 outbox；多节点发送前必须先抢占为 `send_status=3` 并写入 `send_lock_owner/send_lock_expire_time`。
- 若推送超时、网络异常或 5xx，不能重新调用 `nextSeq`，必须复用 outbox 中的 `batch_id/notify_seq/payload_json` 重投。
- 若 `b_wecom_chat_notify_outbox_{ownerShardNo}.uk_notify_seq` 冲突，说明同员工序号分配存在并发或重试 bug；本次事务应回滚并告警，不得自动换新序号掩盖问题。
- 若已有历史 outbox 数据，启用序列表前必须按 `client_id + employee_id` 初始化 `current_seq = max(notify_seq)`，避免从 1 重新分配。

并发容量评估：

- 本方案只串行同一个 `clientId + employeeId` 的通知流，不同员工之间可并发分配 `notifySeq`，因此不会形成全局序号锁。
- 按 5000 万好友容量估算，日通知 batch 与日消息量同阶，约为日消息量的 `1.0-1.3` 倍。高场景 `1.5 亿` 日消息对应 `1.5 亿-1.95 亿` 日通知 batch，平均 `1736-2257 batch/s`，按 10 倍峰值约 `17360-22570 batch/s`。
- 若有 `1 万` 名活跃聚合员工承接高场景流量，单员工 10 倍峰值约 `1.7 batch/s`；即使按 `1000` 名活跃员工的保守集中模型，单员工 10 倍峰值约 `17.4 batch/s`。该量级低于单员工行级锁分配的目标承载区间。
- 生产目标建议单员工通知流持续峰值 `<= 20 batch/s`，短时峰值 `<= 50 batch/s`。超过该阈值时，应优先把同一员工 100-200ms 内的同类视图变更合并到一个 `events[]` batch，而不是把每个字段变化都拆成独立 `notifySeq`。
- 序号分配事务内不得执行工具网关网络调用；`notify/push` 必须在 outbox 提交后异步或同步发起，否则外部网络抖动会放大单员工行锁等待。
- 在 5000 万好友目标容量下，主要瓶颈不是 `b_wecom_chat_notify_seq`，而是 `b_wecom_chat_notify_outbox` 的写入、索引体积、保留期清理和 `notify/pull` 范围查询。正式实现采用 outbox 物理分片，不依赖单表 outbox。

outbox 物理分片设计：

- 逻辑表名仍为 `b_wecom_chat_notify_outbox`，正式物理表为 `b_wecom_chat_notify_outbox_{00..63}`；当前 SQL 已创建 64 个物理分片，未带分片后缀的基表仅作为 `CREATE TABLE ... LIKE` 模板，不作为正式业务写入目标。
- 路由规则固定为 `ownerShardNo = employeeId % 64`。写入 outbox、`notify/pull` 精确拉取、`notify/pull` 范围拉取、重投扫描、保留期清理和排障查询都必须先按 `employeeId` 计算 `ownerShardNo`，只访问对应物理表。
- 每张物理分片表保留同一套字段和索引：`uk_notify_seq(client_id, employee_id, notify_seq, is_deleted)`、`uk_business_event(client_id, employee_id, business_event_key, is_deleted)`、`uk_batch_id(batch_id, is_deleted)`、`idx_employee_seq(client_id, employee_id, notify_seq)`、`idx_send_status(send_status, gmt_modified_time)`、`idx_send_lock(send_status, send_lock_expire_time, gmt_modified_time)`、`idx_retention_expire(retention_expire_time, id)`。全局唯一性由 `employeeId -> ownerShardNo` 固定路由保证，不做跨分片唯一索引。
- `b_wecom_chat_notify_seq` 仍按 `client_id + employee_id` 保存序号行；它的行数等于员工通知流数量，不随通知 batch 保留窗口线性膨胀。后续如果员工序号行自身达到热点或容量瓶颈，再按同一 `ownerShardNo` 拆为 `b_wecom_chat_notify_seq_{00..63}`。
- `notify/pull` 不允许全分片扫描。请求必须包含 `clientId + employeeId`，服务端用 `employeeId` 定位单个 outbox 分片，再按 `notifySeqList` 或 `startNotifySeq/endNotifySeq` 走 `idx_employee_seq`。
- 运行期必须通过 Router 生成物理表名，并通过 Mapper 白名单值对象进入动态 SQL；启动或 readiness 必须预检 64 张物理分片和关键索引。详细设计见 [system-design.md](system-design.md) 的 outbox 路由、Mapper 白名单与分片预检设计。
- 若压测显示 64 分片下单分片索引体积、清理耗时或 `notify/pull` 延迟不可接受，扩容到 128/256 分片。扩容需要发布新路由版本，采用新老分片双读或迁移窗口，不允许直接修改取模导致历史 batch 无法补偿拉取。

生成时机：

```text
业务变更完成
  ↓
组装通知 batch
  ↓
按 clientId + employeeId 获取下一个 notifySeq
  ↓
持久化 outbox：clientId + employeeId + notifySeq + batchId + events[] + status
  ↓
投递给聚合工具网关 relay
```

不要按以下维度生成 `notifySeq`：

| 不推荐维度              | 原因                                                                |
| ----------------------- | ------------------------------------------------------------------- |
| `wecomAccountId`        | 一个员工可管理多个企微账号，最近接待列表、ACK、断线续点都是员工视图 |
| `conversationId`        | 一个 batch 可能同时影响消息、摘要、好友、账号状态等不同对象         |
| `connectionId/deviceId` | 断线重连后连接会变化，续点游标不应绑定一次物理连接                  |

生产约束：

- 同一 `clientId + employeeId` 下必须单调递增，不能回退。
- 已持久化且可重放的 batch 建议连续；如果发生跳号，relay 需要能判断缺失 `notifySeq` 不存在可重放 batch。
- `notifySeq` 生成、batch 组装、outbox 持久化必须作为同一个可靠写入流程处理。
- 生成后持久化失败的 `notifySeq` 不得暴露给 relay 或客户端。
- 同一个业务 batch 重试必须复用原 `notifySeq` 和 outbox 原 `payload_json`，不能重新生成，也不能重投时重新查询当前业务表再组装 payload。
- 命中全局关键词过滤的客户新消息不生成通知 batch，也不分配 `notifySeq`；relay 和客户端不得把这类消息视为通知序号缺口。

`notifySeq` 的三类职责：

| 职责           | 用途                                                                                         | 使用方                 |
| -------------- | -------------------------------------------------------------------------------------------- | ---------------------- |
| 幂等去重键     | 业务后台重投同 batch 时，relay 发现 `clientId + employeeId + notifySeq` 已处理，直接返回成功 | 工具网关 relay         |
| 客户端续点游标 | 客户端重连携带 `sinceNotifySeq`，relay 重放 `notifySeq > sinceNotifySeq` 的 batch            | 客户端、工具网关 relay |
| batch 原子标识 | 一个 batch 内多个事件共享同一 `notifySeq`，客户端作为原子单元应用                            | 客户端、工具网关 relay |

## 3. 业务后台 outbox 规则

业务后台必须落库通知 outbox / event log，保存发给工具网关的 batch。本期逻辑表名为 `b_wecom_chat_notify_outbox`，正式物理表为 `b_wecom_chat_notify_outbox_{00..63}`，该分片表是工具网关通知补偿的权威逻辑盘，relay event log 只是工具网关本地副本。

建议 outbox 逻辑字段：

| 字段                                    | 说明                                                                 |
| --------------------------------------- | -------------------------------------------------------------------- |
| `client_id`                             | 固定 `rh_wxchat`                                                     |
| `employee_id`                           | 目标员工 ID                                                          |
| `notify_seq`                            | `clientId + employeeId` 维度递增序号                                 |
| `batch_id`                              | 默认 `{clientId}:{employeeId}:{notifySeq}`                           |
| `owner_shard_no`                        | 员工分片号，默认 `employeeId % 64`，用于后续水平分片和清理扫描       |
| `protocol_version/source_app/trace_id`  | 通知协议版本、来源应用和链路追踪字段                                 |
| `event_count/event_types`               | 通知事件数量和事件类型摘要                                           |
| `business_event_key`                    | 业务事件幂等键，同一业务语义事件重复执行必须复用                     |
| `source_table/source_id/source_version` | 产生通知的业务事实来源、ID 和版本                                    |
| `payload_json`                          | 完整通知 batch                                                       |
| `send_status`                           | `0 待发送`、`1 已发送`、`2 发送失败`、`3 发送中`                     |
| `send_lock_owner/send_lock_expire_time` | 发送中租约持有者和过期时间，用于多节点重投抢占                       |
| `retry_count`                           | 投递重试次数                                                         |
| `last_send_time/last_ack_time`          | 最近推送时间、工具网关受理成功时间；`last_ack_time` 不代表客户端 ACK |
| `last_error`                            | 最近一次失败原因                                                     |
| `retention_expire_time`                 | 通知 batch 可被补偿拉取的保留截止时间                                |
| `created_time/updated_time`             | 标准时间字段                                                         |

建议唯一键：

```text
uk_notify_seq(client_id, employee_id, notify_seq, is_deleted)
uk_business_event(client_id, employee_id, business_event_key, is_deleted)
uk_batch_id(batch_id, is_deleted)
idx_employee_seq(client_id, employee_id, notify_seq)
idx_send_status(send_status, gmt_modified_time)
idx_send_lock(send_status, send_lock_expire_time, gmt_modified_time)
idx_retention_expire(retention_expire_time, id)
```

业务后台重投规则：

- 网络超时、relay 返回未知结果、relay 暂时不可用时，复用原 `batchId/notifySeq/payload` 重投。
- relay 返回“已处理”也视为成功。
- 不允许因重试生成新的 `notifySeq`，否则客户端会收到重复语义事件。

业务后台补偿拉取规则：

- 工具网关调用 `POST /wechat-business-app/rpc/v1/wecomAggregate/notify/pull` 按 `clientId + employeeId` 从 outbox 拉取通知 batch。
- 支持两种查询模式：指定 `notifySeqList` 精确拉取，或指定 `startNotifySeq/endNotifySeq` 闭区间范围拉取；两种模式二选一。
- 返回的 `payload` 必须是原始 outbox 中的完整通知 batch，与推送给 relay 的 6.3 请求 body 保持一致。
- 指定序号不存在、尚未生成、已过保留期或不属于该 `clientId + employeeId` 时，不伪造空 batch，在 `missingNotifySeqList` 中返回。
- 单次拉取数量建议限制在 `200` 条以内；超过限制时由工具网关分批拉取。
- 补偿拉取不得修改 outbox 的 `notifySeq/batchId/payload_json`，也不得重新计算事件内容。

## 4. 工具网关 relay 规则

relay 必须有事件日志，用于幂等、ACK 和断线重放。

建议 relay event log 逻辑字段：

| 字段            | 说明                                   |
| --------------- | -------------------------------------- |
| `client_id`     | 固定 `rh_wxchat`                       |
| `employee_id`   | 目标员工 ID                            |
| `notify_seq`    | 业务后台生成的通知序号                 |
| `batch_id`      | batch 幂等 ID                          |
| `payload_json`  | 完整通知 batch                         |
| `receive_time`  | relay 接收时间                         |
| `fanout_status` | 推送状态，例如待推送、已推送、部分失败 |

relay 接收 batch 后：

1. 按 `clientId + employeeId + notifySeq` 做幂等判断。
2. 已处理过的 batch 直接返回成功，不重复入库，不重复 fan-out。
3. 未处理过的 batch 先持久化 relay event log，再推送该 `employeeId` 的在线连接。
4. relay 不重新分配 `notifySeq`，不修改 `events[]` 的业务语义。
5. 客户端 ACK 成功后，relay 记录该连接或设备的已确认 `notifySeq`。
6. 客户端重连时携带 `sinceNotifySeq`，relay 重放该员工 `notifySeq > sinceNotifySeq` 的 batch。

如果 relay 本地 event log 不完整或推送接口曾经丢失：

1. relay 先按缺失序号集合或缺失范围调用业务后台 `notify/pull` 补齐 outbox batch。
2. 拉取成功的 batch 仍按 `clientId + employeeId + notifySeq` 幂等写 relay event log。
3. relay event log 补齐后再执行客户端断线重放。
4. 业务后台返回 `missingNotifySeqList` 的序号不得由 relay 自行补号；若客户端仍存在缺口，按静默兜底刷新处理。

客户端重连参数：

```text
sinceNotifySeq = lastNotifySeq
```

如果 relay 通过本地 event log 和业务后台 outbox 都无法完整重放缺失 batch：

- 客户端静默调用 `recentFriends` 兜底刷新接待列表。
- 当前会话打开时，客户端再调用 `message/history` 兜底刷新消息。
- 客户端不得清空列表或展示全屏 loading。

## 5. 外层通知 batch 字段

通知 payload 采用外层 batch + 内层 `events[]` 结构。单事件也必须放入 `events[]`。

| 字段              | 类型   | 必填 | 生成方   | 说明                                       |
| ----------------- | ------ | ---- | -------- | ------------------------------------------ |
| `protocolVersion` | string | 是   | 业务后台 | 协议版本，本期固定 `1.0`                   |
| `notifySeq`       | long   | 是   | 业务后台 | `clientId + employeeId` 维度通知序号       |
| `clientId`        | string | 是   | 业务后台 | 固定 `rh_wxchat`                           |
| `employeeId`      | long   | 是   | 业务后台 | 目标员工 ID                                |
| `batchId`         | string | 是   | 业务后台 | 默认 `{clientId}:{employeeId}:{notifySeq}` |
| `batchTime`       | string | 是   | 业务后台 | batch 生成时间，`yyyy-MM-dd HH:mm:ss`      |
| `sourceApp`       | string | 是   | 业务后台 | 固定 `wechat-business-app`                 |
| `traceId`         | string | 是   | 业务后台 | 链路追踪 ID，用于排障                      |
| `events`          | array  | 是   | 业务后台 | 一个或多个业务事件，按数组顺序原子应用     |

外层示例：

```json
{
  "protocolVersion": "1.0",
  "notifySeq": 1024,
  "clientId": "rh_wxchat",
  "employeeId": 1943599583609966592,
  "batchId": "rh_wxchat:1943599583609966592:1024",
  "batchTime": "2026-05-14 10:30:00",
  "sourceApp": "wechat-business-app",
  "traceId": "TRACE_20260514103000_001",
  "events": []
}
```

## 6. events[] 通用字段

| 字段             | 类型   | 必填         | 说明                                                                                          |
| ---------------- | ------ | ------------ | --------------------------------------------------------------------------------------------- |
| `eventId`        | string | 是           | 建议 `{batchId}:{index}`，用于客户端日志和排查                                                |
| `eventType`      | string | 是           | 事件类型                                                                                      |
| `eventReason`    | string | 是           | 事件原因，不写入业务表                                                                        |
| `conversationId` | string | 会话事件必填 | 会话 ID；无会话事件可为空                                                                     |
| `wecomAccountId` | string | 账号事件必填 | 企微员工账号 ID；无账号事件可为空                                                             |
| `externalUserId` | string | 客户事件必填 | 外部联系人 userid；无客户事件可为空                                                           |
| `eventTime`      | string | 是           | 事件产生时间，`yyyy-MM-dd HH:mm:ss`                                                           |
| 业务快照         | object | 是           | 与 `eventType` 匹配的 `message/sessionSummary/friend/accountBinding/accountStatus/forceClose` |

事件语义必须保持单一：

- `MESSAGE_UPSERT` 只表达消息变更。
- `SESSION_SUMMARY_UPSERT` 只表达最近接待列表摘要变更。
- `FRIEND_UPSERT` 只表达好友和客户资料变更。
- `ACCOUNT_BINDING_CHANGE` 只表达账号权限集合变化。
- `ACCOUNT_STATUS_CHANGE` 只表达账号在线、离线、异常变化。
- `CONNECTION_FORCE_CLOSE` 只表达员工级连接关闭。

同一业务动作同时影响多个视图时，在同一个 batch 的 `events[]` 中放多个事件，共享同一 `notifySeq`。

## 7. 事件类型总表

| eventType                | 业务快照字段     | 职责                                       | 不负责                   |
| ------------------------ | ---------------- | ------------------------------------------ | ------------------------ |
| `MESSAGE_UPSERT`         | `message`        | 当前会话某条消息新增或字段变化             | 不更新最近接待列表       |
| `SESSION_SUMMARY_UPSERT` | `sessionSummary` | 最近接待列表某一行摘要新增或字段变化       | 不更新消息气泡正文       |
| `FRIEND_UPSERT`          | `friend`         | 好友关系、客户资料、跟进信息、标签快照变化 | 不表达摘要排序和未读     |
| `ACCOUNT_BINDING_CHANGE` | `accountBinding` | 员工可管理企微账号集合或绑定权限变化       | 不表达账号在线/离线      |
| `ACCOUNT_STATUS_CHANGE`  | `accountStatus`  | 企微账号在线、离线、异常等运行状态变化     | 不表达员工失去账号权限   |
| `CONNECTION_FORCE_CLOSE` | `forceClose`     | 员工聚合工具长连接需要关闭                 | 不表达某条消息或摘要变化 |

不再单独定义：

| 旧事件                      | 处理方式                      |
| --------------------------- | ----------------------------- |
| `MESSAGE_STATUS_CHANGE`     | 并入 `MESSAGE_UPSERT`         |
| `SESSION_READ`              | 并入 `SESSION_SUMMARY_UPSERT` |
| `CUSTOMER_DETAIL_REFRESHED` | 并入 `FRIEND_UPSERT`          |

## 8. MESSAGE_UPSERT

### 8.1 eventReason

| eventReason                   | 场景                                                      |
| ----------------------------- | --------------------------------------------------------- |
| `CUSTOMER_MESSAGE_RECEIVED`   | 客户消息通过平台回调入库，当前默认平台为 `EASY`           |
| `SEND_PENDING_CREATED`        | 聚合客户端发送接口创建 pending 消息                       |
| `SEND_CONFIRMED`              | 平台确认发送成功，回填平台消息标识和 `sortKey`            |
| `SEND_FAILED`                 | 平台发送失败，更新 `sendStatus=4` 和失败原因              |
| `MESSAGE_REVOKED`             | 平台撤回回调命中消息，更新 `messageStatus=1`              |
| `MESSAGE_DELETED`             | 系统内部屏蔽或删除消息，更新 `messageStatus=2`            |
| `ATTACHMENT_TRANSFER_CHANGED` | 附件转存状态或 OSS 地址变化                               |
| `MULTI_DEVICE_SYNCED`         | 员工官方企微客户端发送，经多端同步入库                    |
| `SYNC_MSG_COMPENSATED`        | 平台历史补偿补入缺失消息，当前 `EASY` 对应 `/msg/syncMsg` |

### 8.2 message 字段

| 字段               | 类型   | 必填 | 说明                                                                   |
| ------------------ | ------ | ---- | ---------------------------------------------------------------------- |
| `localMessageId`   | string | 是   | 本地消息 ID，用于更新或插入消息气泡                                    |
| `requestMessageId` | string | 否   | 客户端发送请求幂等 ID，接收消息和多端同步可为空                        |
| `messageDirection` | int    | 是   | `1` 发送，`2` 接收，`3` 多端同步                                       |
| `messageType`      | int    | 是   | `1` 文本，`2` 图片，`3` 文件，`4` 语音，`5` 图文混合                   |
| `messageStatus`    | int    | 是   | `0` 正常，`1` 撤回，`2` 删除                                           |
| `sendStatus`       | int    | 是   | `0` 无需发送，`1` 待发送，`2` 发送中，`3` 成功，`4` 失败               |
| `sequenceSource`   | string | 是   | `PLATFORM_SEQ`、`LOCAL_PENDING`、`FALLBACK`                            |
| `sortKey`          | string | 是   | 客户端排序和去重更新依据                                               |
| `messageTime`      | string | 是   | `yyyy-MM-dd HH:mm:ss`                                                  |
| `contentText`      | string | 否   | 文本消息正文；图文混合时为文本正文；非文本、撤回、删除或敏感内容可为空 |
| `contentSummary`   | string | 是   | 展示摘要，例如“[图片]”“[文件] 合同.pdf”“文本摘要 [图片]”“消息已撤回”   |
| `failReason`       | string | 否   | 发送失败、转存失败或处理失败原因                                       |
| `attachments`      | array  | 否   | 图片、文件、语音的可展示附件基础信息；图文混合时携带图片附件列表       |

附件字段：

| 字段                 | 类型   | 必填 | 说明                                         |
| -------------------- | ------ | ---- | -------------------------------------------- |
| `attachmentType`     | int    | 是   | `1` 图片，`2` 文件，`3` 语音                 |
| `fileName`           | string | 否   | 文件名                                       |
| `fileSize`           | long   | 否   | 文件大小                                     |
| `durationSeconds`    | int    | 否   | 语音时长                                     |
| `ossFilePath`        | string | 否   | 融合 OSS 文件路径                            |
| `ossPreviewFilePath` | string | 否   | 融合 OSS 预览路径                            |
| `transferStatus`     | int    | 是   | `0` 无需转存，`1` 待转存，`2` 成功，`3` 失败 |

### 8.3 客户端动作

- 按 `localMessageId` upsert 当前会话消息。
- `SEND_CONFIRMED/SEND_FAILED` 必须更新原发送中气泡，不能新增第二条消息。
- 撤回或删除时，按 `messageStatus` 更新原消息展示。
- 字段不足、本地没有该 `localMessageId` 或附件仍在转存中时，调用 `message/history` 兜底。
- 不更新最近接待列表；如果摘要也变化，同 batch 中必须另带 `SESSION_SUMMARY_UPSERT`。
- 客户新消息命中全局关键词过滤时，不发送 `CUSTOMER_MESSAGE_RECEIVED` 的 `MESSAGE_UPSERT`；历史消息仍可由 `message/history` 拉取。

### 9.3 客户端动作

- 按 `conversationId` 定位接待列表项。
- 本地无列表项且 `eventReason=SUMMARY_CREATED` 时插入；本地无列表项但不是新建摘要时，静默调用 `recentFriends`。
- 用服务端 `sessionSummary` 覆盖本地同字段，不在客户端自增自减未读数。
- 按 `lastSortKey/lastMessageTime` 调整列表位置。
- `onlyUnread=true` 且 `hasUnread=false` 时，从当前未读列表移除或静默刷新。
- 不更新消息气泡正文。

## 15. 客户端 ACK 与缺口处理

客户端成功应用 batch 后 ACK：

```json
{
  "clientId": "rh_wxchat",
  "employeeId": 1943599583609966592,
  "clientSessionId": "CS_20260514103000_001",
  "deviceId": "MAC_001",
  "ackNotifySeq": 1024,
  "ackTime": "2026-05-14 10:30:01"
}
```

客户端处理规则：

- 收到 `notifySeq = lastNotifySeq + 1`：按 `events[]` 顺序应用，并在成功后 ACK。
- 收到重复 `notifySeq <= lastNotifySeq`：忽略或仅记录日志，不重复应用。
- 收到 `notifySeq > lastNotifySeq + 1`：认为通知缺口，暂停应用该 batch，优先请求 relay 重放缺口；无法重放时静默调用 `recentFriends/messageHistory`。
- 应用 batch 中任何事件失败：不提升 `lastNotifySeq`，触发兜底刷新。

## 16. 生产约束

- 通知协议字段以本文档为权威，其他文档不得重复定义完整字段表。
- 所有事件必须放入外层 batch 的 `events[]`。
- `eventType/eventReason` 必填，且 `eventReason` 不写入业务表。
- 每个事件必须携带与 `eventType` 匹配的业务快照。
- 同一业务动作同时影响多个视图时，必须放在同一 batch 中共享同一 `notifySeq`。
- 业务后台 outbox 和 relay event log 都必须按 `clientId + employeeId + notifySeq` 幂等。
- 业务后台 outbox 是通知补偿拉取的权威逻辑盘；工具网关 relay event log 缺失时，必须先按 `notifySeqList` 或 `startNotifySeq/endNotifySeq` 调用云端 `notify/pull` 补齐，再决定是否触发客户端兜底刷新。
- `wechat-business-app` 不维护客户端连接，不直接推客户端；聚合工具网关不得信任客户端声明的员工或企微账号身份。
- 通知路由必须以 `employeeId` 为主，`wecomAccountId` 只能做辅助订阅和业务定位。
- 客户端不能按通知到达顺序决定消息顺序；消息顺序仍以 `sortKey` 为准。
- 客户端不能自行推算未读数；未读以 `SESSION_SUMMARY_UPSERT.sessionSummary.unreadCount` 或 `recentFriends` 返回为准。
- 命中过滤关键词的客户消息不会触发未读变化通知，客户端也不应因其他消息通道自行把它计入未读。
- 发送中消息确认后必须复用同一 `localMessageId` 更新原气泡。
- 任何通知应用失败、通知缺口、断线重连，都必须通过业务接口兜底刷新。

涉及消息、摘要、通知和客户端展示的字段变更时，开发阶段必须按“五列对照”检查：

| 检查项     | 必须同步确认                                                         |
| ---------- | -------------------------------------------------------------------- |
| 数据模型   | 表结构、字段默认值、字段注释、索引是否需要变化                       |
| 维护逻辑   | 字段在哪些状态流转中写入、更新、清空或保持不变                       |
| 查询接口   | `recentFriends`、`message/history` 等接口出参是否返回该字段          |
| 网关通知   | `MESSAGE_UPSERT`、`SESSION_SUMMARY_UPSERT` 等 payload 是否携带该字段 |
| 客户端展示 | 客户端能否根据字段完成局部刷新，字段缺失时是否有兜底接口             |

## 17. 验收清单

- 客户新消息：同一 batch 包含 `MESSAGE_UPSERT + SESSION_SUMMARY_UPSERT`，客户端追加消息、置顶会话、未读增加。
- 客户新消息命中过滤关键词：不产生通知 batch；客户端保持当前摘要和未读不变，打开会话后通过历史接口可见该消息。
- 聚合客户端发送成功/失败：同一 `localMessageId` 更新原气泡，摘要 `lastSendStatus` 同步变化。
- `markRead`：只发 `SESSION_SUMMARY_UPSERT`，客户端清除红点；`onlyUnread=true` 时从未读列表移除。
- 客户详情刷新：发 `FRIEND_UPSERT`；如影响摘要冗余字段，同 batch 追加 `SESSION_SUMMARY_UPSERT`。
- 账号离线/异常：发 `ACCOUNT_STATUS_CHANGE`，历史和接待列表仍可见，发送受限。
- 账号禁用/转移：发 `ACCOUNT_BINDING_CHANGE`，刷新账号集合和接待列表。
- 强制下线：发 `CONNECTION_FORCE_CLOSE`，客户端关闭连接并按原因清理登录态。
- 断线重连：客户端携带 `sinceNotifySeq`，relay 先用本地 event log 重放缺失 batch；本地缺失时调用业务后台 `notify/pull` 从 outbox 补齐；仍无法重放时客户端静默刷新。
