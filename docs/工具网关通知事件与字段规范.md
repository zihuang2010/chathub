# 工具网关通知事件与字段规范

本文档定义 `wechat-business-app` 通知聚合工具网关、再由工具网关通过 HTTP/2 gRPC 长连接推送客户端的通知协议。客户接待列表同步方案见同目录 [客户接待列表轻量同步方案.md](客户接待列表轻量同步方案.md)，接待列表业务维护机制见 [客户接待列表与网关通知机制.md](客户接待列表与网关通知机制.md)。

## 1. 协议边界

- 业务后台负责生成业务通知 batch，并持久化通知 outbox / event log。
- 聚合工具网关作为 relay，负责接收通知 batch、幂等入库、fan-out 推送、ACK 记录和断线重放。
- 客户端只和聚合工具网关保持一个员工级 HTTP/2 gRPC 长连接。
- 本期连接模型为一个员工客户端会话一个长连接、一个订阅；客户端收到员工级通知后，按 `customerUserId/conversationId` 自行 fan-out 到对应企微账号页签、接待列表项或会话窗口。
- 工具网关不得信任客户端声明的 `employeeId/customerUserId`，连接身份必须来自 `verifyToken` 返回值。

## 2. notifySeq 生成规则与 scope

`notifySeq` 由业务后台生成，不由聚合工具网关重新编号。

推荐 scope：

```text
clientId + employeeId
```

本期固定 `clientId = rh_wxchat`，因此实现上可理解为：

```text
employeeId 维度单调递增
```

生成规则：

```text
notifySeq = nextSeq(clientId, employeeId)
```

生成时机：

```text
业务变更完成
  ↓
组装通知 batch
  ↓
按 clientId + employeeId 获取下一个 notifySeq
  ↓
将 notifySeq + employeeId + events[] 持久化到 outbox/event_log
  ↓
投递给聚合工具网关 relay
```

`notifySeq` 不按以下维度生成：

| 不推荐维度              | 原因                                                              |
| ----------------------- | ----------------------------------------------------------------- |
| `customerUserId`        | 一个员工管理多个企微账号，最近接待列表、ACK、断线续点都是员工视图 |
| `conversationId`        | 一个通知 batch 可能同时影响多个会话或账号集合                     |
| `connectionId/deviceId` | 断线重连后连接会变化，续点游标不应绑定一次物理连接                |

单调性要求：

- 同一 `clientId + employeeId` 下必须单调递增，可以有跳跃，不能回退。
- 已持久化并可被 relay 重放的通知 batch，建议尽量连续。
- 如果生成后持久化失败，该 `notifySeq` 不应暴露给 relay 或客户端。
- 如果确实发生跳号，relay 需要能判断缺失 `notifySeq` 不存在可重放 batch；客户端发现缺口后仍按轻量方案静默刷新兜底。
- 业务后台重投同一个 batch 时必须复用原 `notifySeq`，不能重新生成。

`notifySeq` 同时承担三类职责：

| 职责           | 用途                                                                                                                | 使用方                 |
| -------------- | ------------------------------------------------------------------------------------------------------------------- | ---------------------- |
| 幂等去重键     | 网络抖动或业务后台重投同 batch 时，relay 发现该 `employeeId + notifySeq` 已处理，直接返回成功，不重复入库或 fan-out | 工具网关 relay         |
| 客户端续点游标 | 客户端断线重连时携带 `sinceNotifySeq`，relay 重放该员工 `notifySeq > sinceNotifySeq` 的 batch                       | 客户端、工具网关 relay |
| Batch 原子标识 | 一个 batch 内多个事件语义上是一个原子单元，共享同一 `notifySeq`，客户端要么一起收到，要么靠续点一起补回             | 客户端、工具网关 relay |

## 3. 外层通知包字段

通知 payload 采用外层通知包 + 内层 `events[]` 结构。单事件也必须放入 `events[]`，不能退化为单对象。

| 字段              | 类型   | 必填 | 生成方   | 说明                                                                    |
| ----------------- | ------ | ---- | -------- | ----------------------------------------------------------------------- |
| `notifySeq`       | long   | 是   | 业务后台 | `clientId + employeeId` 维度的通知 batch 序号                           |
| `clientId`        | string | 是   | 业务后台 | 固定 `rh_wxchat`                                                        |
| `employeeId`      | long   | 是   | 业务后台 | 目标员工 ID                                                             |
| `batchId`         | string | 建议 | 业务后台 | 通知 batch 幂等 ID，便于排查；可由 `clientId:employeeId:notifySeq` 派生 |
| `batchTime`       | string | 是   | 业务后台 | batch 生成时间，`yyyy-MM-dd HH:mm:ss`                                   |
| `clientSessionId` | string | 否   | 工具网关 | 工具网关推送时可补充的客户端会话 ID                                     |
| `deviceId`        | string | 否   | 工具网关 | 工具网关推送时可补充的设备 ID                                           |
| `events`          | array  | 是   | 业务后台 | 一个或多个业务事件，按数组顺序原子应用                                  |

外层示例：

```json
{
  "notifySeq": 1024,
  "clientId": "rh_wxchat",
  "employeeId": 1943599583609966592,
  "batchId": "rh_wxchat:1943599583609966592:1024",
  "batchTime": "2026-05-14 10:30:00",
  "clientSessionId": "CS_20260514103000_001",
  "deviceId": "MAC_001",
  "events": []
}
```

## 4. Relay 处理规则

聚合工具网关 relay 收到业务后台通知 batch 后：

1. 按 `clientId + employeeId + notifySeq` 做幂等判断。
2. 已处理过的 batch 直接返回成功，不重复写入事件日志，不重复 fan-out。
3. 未处理过的 batch 先持久化到 relay 事件日志，再推送该 `employeeId` 的在线连接。
4. relay 不重新分配 `notifySeq`，也不修改 `events[]` 的业务语义。
5. 客户端 ACK 成功后，relay 记录该连接或设备的已确认 `notifySeq`。
6. 客户端重连时携带 `sinceNotifySeq`，relay 重放该员工 `notifySeq > sinceNotifySeq` 的 batch。

客户端重连建议携带：

```text
sinceNotifySeq = lastNotifySeq
```

如果 relay 无法完整重放缺失 batch，客户端必须静默调用 `recentFriends`，当前会话打开时再调用 `message/history` 兜底。

## 5. events[] 通用字段

`events[]` 中每个事件至少包含：

| 字段             | 类型   | 必填             | 说明                                                                                          |
| ---------------- | ------ | ---------------- | --------------------------------------------------------------------------------------------- |
| `eventType`      | string | 是               | 事件类型                                                                                      |
| `eventReason`    | string | 是               | 事件原因，不写入业务表，只用于客户端分支、日志和排查                                          |
| `conversationId` | string | 会话事件必填     | 会话 ID；无会话事件可为空                                                                     |
| `customerUserId` | string | 账号相关事件必填 | 客服企微用户 ID；无账号事件可为空                                                             |
| `externalUserId` | string | 客户相关事件必填 | 外部联系人 userid；无客户事件可为空                                                           |
| `eventTime`      | string | 建议             | 事件产生时间，`yyyy-MM-dd HH:mm:ss`                                                           |
| 业务快照         | object | 是               | 与 `eventType` 匹配的 `message/sessionSummary/friend/accountBinding/accountStatus/forceClose` |

事件语义必须保持单一：

- `MESSAGE_UPSERT` 只表达消息变更，不更新最近接待列表。
- `SESSION_SUMMARY_UPSERT` 只表达接待列表摘要变更，不更新消息气泡正文。
- 同一业务动作同时影响消息窗口和接待列表时，在同一个 batch 的 `events[]` 中放两条事件，共享同一 `notifySeq`。

## 6. 事件类型总表

| eventType                | 业务快照字段     | 职责                                       | 不负责                   |
| ------------------------ | ---------------- | ------------------------------------------ | ------------------------ |
| `MESSAGE_UPSERT`         | `message`        | 当前会话某条消息新增或字段变化             | 不更新最近接待列表       |
| `SESSION_SUMMARY_UPSERT` | `sessionSummary` | 最近接待列表某一行摘要新增或字段变化       | 不更新消息气泡正文       |
| `FRIEND_UPSERT`          | `friend`         | 好友关系、客户资料、跟进信息、标签快照变化 | 不表达摘要排序和未读     |
| `ACCOUNT_BINDING_CHANGE` | `accountBinding` | 员工可管理企微账号集合或绑定权限变化       | 不表达账号在线/离线      |
| `ACCOUNT_STATUS_CHANGE`  | `accountStatus`  | 企微账号在线、离线、异常等运行状态变化     | 不表达员工失去账号权限   |
| `CONNECTION_FORCE_CLOSE` | `forceClose`     | 员工聚合工具长连接需要关闭                 | 不表达某条消息或摘要变化 |

不再单独定义：

| 旧事件                      | 处理方式                      |
| --------------------------- | ----------------------------- |
| `MESSAGE_STATUS_CHANGE`     | 并入 `MESSAGE_UPSERT`         |
| `SESSION_READ`              | 并入 `SESSION_SUMMARY_UPSERT` |
| `CUSTOMER_DETAIL_REFRESHED` | 并入 `FRIEND_UPSERT`          |

## 7. MESSAGE_UPSERT

### 7.1 触发原因

| eventReason                   | 场景                                           |
| ----------------------------- | ---------------------------------------------- |
| `CUSTOMER_MESSAGE_RECEIVED`   | 客户消息通过 Easy 回调入库                     |
| `SEND_PENDING_CREATED`        | 聚合客户端发送接口创建 pending 消息            |
| `SEND_CONFIRMED`              | Easy 确认发送成功，回填 Easy 字段和 `sortKey`  |
| `SEND_FAILED`                 | Easy 发送失败，更新 `sendStatus=4` 和失败原因  |
| `MESSAGE_REVOKED`             | Easy 撤回回调命中消息，更新 `messageStatus=1`  |
| `MESSAGE_DELETED`             | 系统内部屏蔽或删除消息，更新 `messageStatus=2` |
| `ATTACHMENT_TRANSFER_CHANGED` | 附件转存状态或 OSS 地址变化                    |
| `MULTI_DEVICE_SYNCED`         | 员工官方企微客户端发送，经多端同步入库         |
| `SYNC_MSG_COMPENSATED`        | Easy `/msg/syncMsg` 历史补偿补入缺失消息       |

### 7.2 message 字段

| 字段               | 类型   | 必填 | 说明                                                     |
| ------------------ | ------ | ---- | -------------------------------------------------------- |
| `localMessageId`   | string | 是   | 本地消息 ID，用于更新或插入消息气泡                      |
| `requestMessageId` | string | 否   | 客户端发送请求幂等 ID，接收消息和多端同步可为空          |
| `messageDirection` | int    | 是   | `1` 发送，`2` 接收，`3` 多端同步                         |
| `messageType`      | int    | 是   | `1` 文本，`2` 图片，`3` 文件，`4` 语音                   |
| `messageStatus`    | int    | 是   | `0` 正常，`1` 撤回，`2` 删除                             |
| `sendStatus`       | int    | 是   | `0` 无需发送，`1` 待发送，`2` 发送中，`3` 成功，`4` 失败 |
| `sequenceSource`   | string | 是   | `EASY_SEQ`、`LOCAL_PENDING`、`FALLBACK`                  |
| `sortKey`          | string | 是   | 客户端排序和去重更新依据                                 |
| `messageTime`      | string | 是   | `yyyy-MM-dd HH:mm:ss`                                    |
| `contentText`      | string | 否   | 文本消息正文；非文本、撤回、删除或敏感内容可为空         |
| `contentSummary`   | string | 是   | 展示摘要，例如“[图片]”“[文件] 合同.pdf”“消息已撤回”      |
| `failReason`       | string | 否   | 发送失败、转存失败或处理失败原因                         |
| `attachments`      | array  | 否   | 图片、文件、语音的可展示附件基础信息                     |

附件字段：

| 字段                 | 类型   | 必填 | 说明                                         |
| -------------------- | ------ | ---- | -------------------------------------------- |
| `attachmentType`     | int    | 是   | `1` 图片，`2` 文件，`3` 语音                 |
| `fileName`           | string | 否   | 文件名                                       |
| `fileSize`           | long   | 否   | 文件大小                                     |
| `durationSeconds`    | int    | 否   | 语音时长                                     |
| `ossFilePath`        | string | 否   | 融合 OSS 文件路径                            |
| `ossPreviewFilePath` | string | 否   | 融合 OSS 预览路径                            |
| `transferStatus`     | int    | 是   | `0` 无需转存，`1` 待转存，`2` 成功，`3` 失败 |

### 7.3 客户端动作

- 按 `localMessageId` upsert 当前会话消息。
- `SEND_CONFIRMED/SEND_FAILED` 必须更新原发送中气泡，不能新增第二条消息。
- 撤回或删除时，按 `messageStatus` 更新原消息展示。
- 通知字段不足、本地没有该 `localMessageId` 或附件仍在转存中时，调用 `message/history` 兜底。
- 不更新最近接待列表；如果摘要也变化，同一 batch 中必须另带 `SESSION_SUMMARY_UPSERT`。

## 8. SESSION_SUMMARY_UPSERT

### 8.1 触发原因

| eventReason                | 场景                                       |
| -------------------------- | ------------------------------------------ |
| `SUMMARY_CREATED`          | 新会话首次产生摘要                         |
| `LAST_MESSAGE_CHANGED`     | 最后一条消息、时间或排序变化               |
| `LAST_SEND_STATUS_CHANGED` | 最后一条消息发送状态变化                   |
| `UNREAD_CHANGED`           | 未读数变化                                 |
| `MARK_READ`                | `markRead` 成功，清除未读                  |
| `SUMMARY_PROFILE_CHANGED`  | 好友资料或企微账号别名导致摘要冗余字段变化 |

### 8.2 sessionSummary 字段

| 字段                   | 类型    | 必填 | 说明                                                                 |
| ---------------------- | ------- | ---- | -------------------------------------------------------------------- |
| `lastLocalMessageId`   | string  | 是   | 摘要最后一条消息 ID                                                  |
| `lastMessageType`      | int     | 是   | 最后一条消息类型                                                     |
| `lastMessageDirection` | int     | 是   | 最后一条消息方向                                                     |
| `lastSendStatus`       | int     | 是   | 最后一条消息发送状态：0 无需发送，1 待发送，2 发送中，3 成功，4 失败 |
| `lastMessageSummary`   | string  | 否   | 最后一条消息摘要                                                     |
| `lastMessageTime`      | string  | 是   | 最后一条消息时间，`yyyy-MM-dd HH:mm:ss`                              |
| `lastSortKey`          | string  | 是   | 摘要排序键，客户端按服务端顺序调整列表位置                           |
| `unreadCount`          | int     | 是   | 当前员工维度未读数                                                   |
| `hasUnread`            | boolean | 是   | 是否展示未读红标                                                     |
| `externalName`         | string  | 否   | 客户名称，资料变化时可返回                                           |
| `externalAvatar`       | string  | 否   | 客户头像，资料变化时可返回                                           |
| `wecomAlias`           | string  | 否   | 企微账号展示别名，账号别名变化时可返回                               |

### 8.3 客户端动作

- 按 `conversationId` 定位接待列表项。
- 本地无列表项且 `eventReason=SUMMARY_CREATED` 时插入；本地无列表项但不是新建摘要时，静默调用 `recentFriends` 兜底。
- 用服务端 `sessionSummary` 覆盖本地同字段，不在客户端自增自减未读数。
- 按 `lastSortKey/lastMessageTime` 调整列表位置。
- `onlyUnread=true` 且 `hasUnread=false` 时，从当前未读列表移除或静默刷新。
- 不更新消息气泡正文。

## 9. FRIEND_UPSERT

### 9.1 触发原因

| eventReason             | 场景                                     |
| ----------------------- | ---------------------------------------- |
| `FRIEND_CREATED`        | 新好友关系产生                           |
| `BASIC_PROFILE_CHANGED` | 好友名称、头像、手机号等基础资料变化     |
| `DETAIL_REFRESHED`      | 客户详情按需或手动刷新成功               |
| `FOLLOW_INFO_CHANGED`   | 跟进备注、描述、备注手机号等跟进信息变化 |
| `TAGS_CHANGED`          | 标签快照变化                             |

### 9.2 friend 字段

| 字段             | 类型   | 必填 | 说明                                        |
| ---------------- | ------ | ---- | ------------------------------------------- |
| `customerUserId` | string | 是   | 客服企微用户 ID                             |
| `externalUserId` | string | 是   | 外部联系人 userid                           |
| `externalName`   | string | 否   | 客户名称                                    |
| `externalAvatar` | string | 否   | 客户头像                                    |
| `externalType`   | int    | 否   | 客户类型，按企微返回值保存                  |
| `externalGender` | int    | 否   | 性别，按企微返回值保存                      |
| `remark`         | string | 否   | 当前跟进关系备注                            |
| `description`    | string | 否   | 当前跟进关系描述                            |
| `remarkMobiles`  | array  | 否   | 备注手机号数组                              |
| `addWay`         | int    | 否   | 添加来源                                    |
| `tags`           | array  | 否   | 标签快照数组                                |
| `lastSyncTime`   | string | 否   | 客户详情最近刷新时间，`yyyy-MM-dd HH:mm:ss` |

### 9.3 客户端动作

- 好友列表已加载时，按 `customerUserId + externalUserId` 局部新增或更新。
- 当前资料面板打开时，更新客户主体、跟进信息和标签快照。
- 如果资料变化影响接待列表冗余字段，同一 batch 追加 `SESSION_SUMMARY_UPSERT`。

## 10. ACCOUNT_BINDING_CHANGE

### 10.1 触发原因

| eventReason             | 场景                   |
| ----------------------- | ---------------------- |
| `ACCOUNT_ADDED`         | 员工新增可管理企微账号 |
| `ACCOUNT_DISABLED`      | 单个企微账号绑定禁用   |
| `ACCOUNT_TRANSFERRED`   | 企微账号转移到其他员工 |
| `ACCOUNT_ALIAS_CHANGED` | 企微账号别名变化       |

### 10.2 accountBinding 字段

| 字段                 | 类型   | 必填 | 说明                            |
| -------------------- | ------ | ---- | ------------------------------- |
| `customerUserId`     | string | 是   | 企微账号 userid                 |
| `ownerEmployeeId`    | long   | 是   | 当前归属员工 ID                 |
| `previousEmployeeId` | long   | 否   | 转移前员工 ID，仅转移场景返回   |
| `wecomName`          | string | 否   | 企微账号名称                    |
| `wecomAccount`       | string | 否   | 企微账号                        |
| `wecomAlias`         | string | 否   | 客户端展示别名                  |
| `bindStatus`         | int    | 是   | 绑定状态：0 禁用，1 启用        |
| `configStatus`       | int    | 是   | 配置状态：0 禁用，1 启用        |
| `changedTime`        | string | 是   | 变化时间，`yyyy-MM-dd HH:mm:ss` |

### 10.3 客户端动作

- 刷新账号列表和订阅集合。
- 刷新最近接待列表。
- 已失去权限的当前会话需要关闭或提示无权限。
- `ACCOUNT_DISABLED/ACCOUNT_TRANSFERRED` 不等于员工连接强制关闭；除非另有 `CONNECTION_FORCE_CLOSE`。

## 11. ACCOUNT_STATUS_CHANGE

### 11.1 触发原因

| eventReason              | 场景                          |
| ------------------------ | ----------------------------- |
| `EASY_STATUS_CALLBACK`   | Easy `cmd=11016` 账号状态回调 |
| `ONLINE_CHECK`           | 后台按需或定时在线状态检查    |
| `SEND_FAILURE_WRITEBACK` | 发送失败反写账号离线或异常    |
| `SEND_SUCCESS_RECOVERED` | 发送成功辅助恢复在线          |

### 11.2 accountStatus 字段

| 字段                 | 类型    | 必填 | 说明                                               |
| -------------------- | ------- | ---- | -------------------------------------------------- |
| `customerUserId`     | string  | 是   | 企微账号 userid                                    |
| `accountStatus`      | int     | 是   | 账号状态：1 在线，2 离线，3 异常                   |
| `statusReason`       | string  | 否   | 状态原因，例如 `cmd=11016`、在线检查、发送失败反写 |
| `statusTime`         | string  | 是   | 状态更新时间，`yyyy-MM-dd HH:mm:ss`                |
| `sendDisabled`       | boolean | 是   | 当前账号是否应禁用发送                             |
| `sendDisabledReason` | string  | 否   | 禁用发送提示文案                                   |

### 11.3 客户端动作

- 刷新账号列表中的在线、离线、异常状态。
- 刷新当前 `customerUserId` 对应账号区域的状态提示。
- `accountStatus=2/3` 时禁用或拦截发送按钮。
- 不隐藏最近接待列表和历史消息；隐藏权限由 `ACCOUNT_BINDING_CHANGE` 或 `CONNECTION_FORCE_CLOSE` 控制。

## 12. CONNECTION_FORCE_CLOSE

### 12.1 触发原因

| eventReason            | 场景                                      |
| ---------------------- | ----------------------------------------- |
| `EXCLUSIVE_LOGIN`      | `rh_wxchat` 排他登录，新 token 踢下旧连接 |
| `EMPLOYEE_DISABLED`    | 员工融合账号被禁用                        |
| `CONFIG_DISABLED`      | 员工聚合聊天配置禁用                      |
| `ACCESS_REVOKED`       | 员工不再满足聚合聊天准入条件              |
| `TOKEN_RECHECK_FAILED` | 工具网关存活复核失败                      |

### 12.2 forceClose 字段

| 字段              | 类型    | 必填 | 说明                                              |
| ----------------- | ------- | ---- | ------------------------------------------------- |
| `closeScope`      | string  | 是   | 关闭范围：`EMPLOYEE` 表示关闭员工所有聚合工具连接 |
| `reasonCode`      | string  | 是   | 断连原因，和 `eventReason` 保持一致               |
| `reasonMessage`   | string  | 是   | 给客户端展示或记录的断连说明                      |
| `clearLocalToken` | boolean | 是   | 是否清理本地 token                                |
| `reloginRequired` | boolean | 是   | 是否需要重新登录                                  |
| `closeTime`       | string  | 是   | 关闭时间，`yyyy-MM-dd HH:mm:ss`                   |

### 12.3 客户端动作

- 根据断开原因提示重新登录、被挤下线或联系管理员。
- 清理本地敏感缓存。
- 关闭当前长连接，不再继续使用旧 token 重连。
